# 项目亮点

- 极度精简，完整实现不到 200 行代码
- 使用简单，提供与 usState 相近的用法，极低的学习和使用成本
- 灵活高效，支持使用 hook 和在组件外任意位置，访问/修改全局状态
- 专注性能，支持使用 selector 和 consumer，更细粒度的控制组件 re-render

# 幕后花絮

React 生态内的状态管理方案已经有很多了，比如：Redux、Mobx、Dva、Zustand、Recoil 等。

这里为什么我还要再写一个状态管理库出来，是为了重复造轮子的酸爽，还是单纯为了炫技，难道社区已有的状态管理库满足不了需求吗？

很显然，都不是。

准确的说，状态管理的核心不过是为了实现跨组件的状态共享，以及状态更新时依赖组件的自动刷新，只要满足这两点，所有的状态管理库都可视为是等价的。

在我过去的一些项目中，也尝试过各种各样的状态管理工具，但我觉得用起来都不是很称手：

有的要么要写很多模版代码，用起来很繁琐，不够灵活；有的要么写起来需要学习新的范式，使用时会多一些额外的心智成本。

XSta 是结合了我对状态管理的深刻理解，并且在过往的大量项目经验和业务实践中，沉淀出的一些最精华的部分。

它足够精简，源码不到 200 行，却拥有相对完备的状态管理能力。

这种精简的背后，是一种高度的取舍，和对业务需求的深刻理解：

它通过 10% 的体积实现了 99% 的项目状态管理的需要，

剩下的 1% 的 usecase（比如使用中间件持久化状态到本地等），是否有必要通过 90% 额外的复杂度和体积来达到？

XSta 的选择是：~砍掉~

总的来说，如果你需要的是一个足够精简，使用方便（不需要复杂的模版代码和额外的学习成本与使用心智），并且久经考验，满足绝大多数项目的状态管理需要的状态管理库，那么 XSta 可能是最佳选择之一。

如果你追求的是绝对的信誉（社区流行度）和大而全的东西，并且对项目尺寸不那么敏感，习惯了之前的使用范式和心智模型，那么你应该继续使用自己熟悉的状态管理库。
